02 העיקרון המארגן

למה בכלל נוצרו מערכות מרובות סוכנים?

רבים ניגשים ל־Claude Code מתוך מחשבה שמדובר בכלי לכתיבת קוד, ולכן קל לראות רכיבים כמו Subagents, Skills ו־Workflows כאוסף פיצ'רים. בפועל, הם אינם פיצ'רים — הם פתרון לבעיה יסודית הרבה יותר. הפרק הזה לא עוסק ב"איך" אלא ב"למה": מי שמבין את העיקרון הזה יקרא את שאר המדריך אחרת.

הבעיה: מורכבות

ככל שהמשימה גדלה, כך קשה יותר ל־Agent יחיד:

  • להחזיק את כל ההקשר בזיכרון.
  • להבין מספר תחומים במקביל.
  • לשמור על עקביות לאורך זמן.
  • לבצע כמה תהליכים בו־זמנית.
  • לבקר את עצמו תוך כדי ביצוע.

וזו הנקודה המהותית: גם אם חלון ההקשר גדול מאוד, עצם ריכוז כל האחריות במקום אחד הופך את החשיבה לפחות יעילה. זה אינו רק אילוץ טכני של מספר טוקנים, אלא מגבלה כללית של ניהול מורכבות — ובפרק 05 נפגוש אותה גם במספרים: "ריקבון הקשר", שבו האיכות דועכת הרבה לפני שהחלון מתמלא.

תזמור דינמי: התהליך נבנה לפי הבעיה

כאשר המשימה פשוטה — Agent יחיד מספיק, וזו גם הבחירה הנכונה. כשהיא מורכבת, המערכת מחליטה באופן דינמי: אילו מומחים דרושים, באיזה סדר יעבדו, אילו תוצאות יעברו לביקורת, וכיצד לאחד את התוצרים. במקום להגדיר מראש תהליך קשיח, התהליך עצמו נבנה בהתאם לבעיה. את העיקרון הזה נפגוש בשלוש רמות הולכות ומתרחבות — subagent בודד, צוות סוכנים, ולבסוף הפיצ'ר שנושא את השם הזה ממש: Dynamic Workflows (פרק 12), שבו Claude כותב בעצמו את קוד התזמור.

אותו עיקרון, בכל מערכת מורכבת

הרעיון אינו ייחודי ל־Claude Code. בכל מערכת מורכבת — גוף האדם, חברה, ארגון או מערכת הפעלה — מגיע שלב שבו רכיב יחיד כבר אינו מסוגל להתמודד עם כל המורכבות. בשלב הזה נוצרת מערכת: חלוקת אחריות, התמחות, תיאום, משוב ובקרה. זהו עיקרון ותיק בתורת המערכות — הרברט סיימון תיאר אותו כבר ב־1962 במאמר "The Architecture of Complexity": מערכות מורכבות ששורדות בנויות כמעט תמיד כהיררכיה של תת־מערכות כמעט־עצמאיות. Claude Code מיישם את אותו עיקרון על פתרון בעיות באמצעות AI.

פקודת /effort שולטת בכמה "מחשבה" Claude משקיע. ככל שגבוה יותר — חכם יותר אך איטי ויקר יותר:

🆕 רמות המאמץ חלות כעת גם על Sonnet 5: ברמת xhigh הוא מתקרב לביצועי Opus 4.8 ברמות medium–high במבחני computer-use וחיפוש אגנטי — אבל בעלות דומה ואף גבוהה יותר, כך ש־Opus נשאר הבחירה הנכונה לחלק מהמשימות. עם השקת Sonnet 5 גם הועלו מגבלות הקצב ב־Claude Code כדי להכיל רמות effort גבוהות.

04 המודל המנטלי

הלולאה האגנטית ושליטה בה

לפני שצוללים לפיצ'רים, חשוב להפנים את הלולאה שמניעה כל פעולה של Claude Code. ברגע שמבינים אותה, כל שאר הכלים (subagents, hooks, plan mode) הופכים לדרכים לעצב או לשלוט בלולאה הזו.

ארבעת השלבים

1

איסוף הקשר

Claude קורא קבצים, מחפש בקוד (grep/glob), בודק git, ומריץ פקודות כדי להבין את המצב — לרוב בעזרת הסוכן המובנה Explore.

2

תכנון

הסוכן מנסח אסטרטגיה. ב־Plan Mode הוא מציג לך תוכנית מלאה לפני שנוגע בקוד.

3

פעולה

עריכת קבצים, הרצת פקודות, התקנת חבילות — כל פעולה כפופה למצב ההרשאות שהגדרת.

4

אימות

הרצת טסטים/לינטר, קריאת הפלט, ותיקון עצמי. הלולאה חוזרת עד שהיעד מושג.

תפקידך כמנהל הוא בעיקר בשלבים 2 ו־4: לאשר תוכנית טובה, ולוודא שהאימות אמיתי (ולא "אמרתי שעובד").

מצב ההרשאה קובע כמה אוטונומיה יש לסוכן. זו ההחלטה הבטיחותית החשובה ביותר שלך:

Plan Mode — הכלי הכי חשוב למתחילים מתקדמים

לחיצה על Shift+Tab (או /plan) מכניסה למצב תכנון: Claude חוקר את הקוד בקריאה בלבד ומציג תוכנית מפורטת. אתה יכול לערוך אותה בעורך לפני אישור. רק אחרי שאתה מאשר — הוא מתחיל לעבוד.

💡"תכנן לפני שתבצע" — הרגל מספר 1

זה אולי הטיפ היחיד שמשנה הכי הרבה. תכנון מראש הופך את הסוכן לצפוי ובטוח להרצה ארוכה ללא פיקוח צמוד. צוותים מדווחים שזה ההבדל בין "כלי שעוזר" ל"כלי שאפשר להשאיר לרוץ".

גלגל אחורה: /rewind

טעה? לחיצה כפולה על Esc (או /rewind) מגלגלת את השיחה ואת הקבצים לנקודת checkpoint קודמת. 🆕 מאז יוני, /rewind יודע לשחזר גם שיחה מלפני /clear, ותפריט ה־Rewind מציע "סכם עד כאן" לדחיסת הקשר מוקדמת. זו רשת ביטחון שמאפשרת לנסות גישות בלי לפחד.

🛠️תרגול
  1. בקש מ־Claude משימה בינונית (למשל "הוסף ולידציה לטופס ההרשמה"), אבל קודם הכנס ל־Plan Mode וקרא את התוכנית.
  2. ערוך את התוכנית — הוסף דרישה שהוא פספס — ואז אשר.
  3. אחרי שסיים, נסה /rewind וחזור למצב לפני העריכה.
05 הנדסת הקשר

הנדסת הקשר וזיכרון

זה הפרק החשוב ביותר במדריך. איכות העבודה של Claude Code נקבעת יותר מכל דבר אחר על ידי מה נמצא בחלון ההקשר. כל פיצ'ר אגנטי שנלמד בהמשך — subagents, workflows — קיים בעיקר כדי לנהל את ההקשר נכון.

הבעיה: "ריקבון הקשר" (Context Rot)

לחלון ההקשר יש גבול — וביולי 2026 הוא זז: Sonnet 5, ברירת המחדל החדשה, מגיע עם 1M טוקנים נטיבית בכל התוכניות (במודלים אחרים: 200K בסטנדרט, 1M לפי תוכנית ומודל). אבל הנה ההפתעה: ככל שההקשר מתמלא, האיכות יורדת עוד הרבה לפני שמגיעים לגבול. הסוכן מתבלבל, שוכח החלטות, ומתחיל לסתור את עצמו. משתמשים מנוסים שומרים על ניצול נמוך:

📌כלל ה־40% / 60%

שאף לשמור את ההקשר מתחת ל־40% ניצול במהלך עבודה אקטיבית, וסגור משימה עד 60%. מעבר לזה — האיכות מתחילה לדעוך. פקודת /context מציגה פילוח טוקנים מדויק.

🆕
הערת השוואה: ה־tokenizer של Sonnet 5 מפיק כ־30% יותר טוקנים לאותו טקסט לעומת קודמיו — אל תשוו מספרי /context בין מודלים אחד־לאחד. כלל ה־40% נמדד באחוזים, ולכן נשאר תקף כמו שהוא.

⚠️הטעות מספר 1: "סשן מטבח" (Kitchen Sink)

לערבב משימות לא קשורות באותו סשן מזהם את ההקשר ומוריד איכות. תיקנת באג? עברת לפיצ'ר חדש? עשה /clear. זול, מהיר, ומחזיר את הסוכן לחדות מלאה. בנוסף — אם ניסית לתקן כיוון שגוי פעמיים ונכשלת, אל תערום תיקונים: /clear או /rewind, ונסח מחדש את הבקשה הראשונית.

CLAUDE.md — הזיכרון הקבוע של הפרויקט

קובץ CLAUDE.md נטען אוטומטית בתחילת כל סשן ושורד /compact. כאן מתעדים את מה שהסוכן חייב לדעת על הפרויקט: מוסכמות קוד, פקודות הרצה, ארכיטקטורה, ו"אל תעשה". יש כמה רמות:

  • שורש הפרויקט — ./CLAUDE.md, מוכנס מחדש גם אחרי compaction.
  • רמת המשתמש — ~/.claude/CLAUDE.md, מוסכמות אישיות לכל הפרויקטים.
  • תיקיות מקוננות — CLAUDE.md בתת־תיקייה נטען כשעובדים בה.

צור שלד התחלתי בפקודה /init. הנה דוגמה תמציתית וטובה:

## פקודות
- התקנה: `pnpm install`
- בדיקות: `pnpm test` (חובה להריץ לפני סיום משימה)
- לינט: `pnpm lint --fix`

## אל תעשה

⚠️אל תנפח את ה־CLAUDE.md

הקובץ הזה עולה טוקנים בכל סשן מחדש. שמור אותו עובדתי וקצר (כ־200 שורות לכל היותר). הסעיף "אל תעשה" עם "השתמש בגישה הפשוטה ביותר" פותר טעות נפוצה נוספת — Claude נוטה להוסיף אבסטרקציות ומורכבות שלא ביקשת.

זיכרון אוטומטי

מעבר ל־CLAUDE.md, Claude כותב לעצמו הערות זיכרון מתוך התיקונים שלך — ולומד לאורך זמן. אפשר לעיין ולערוך אותן עם /memory. subagent יכול לקבל זיכרון משלו דרך השדה memory (נראה בפרק 07).

🛠️תרגול
  1. בפרויקט קיים הרץ /init וערוך את ה־CLAUDE.md שנוצר — הוסף את פקודות הטסט והלינט האמיתיות.
  2. במהלך עבודה, הרץ /context ובדוק מה תופס הכי הרבה טוקנים.
  3. תרגל את ההרגל: בסיום משימה — /clear לפני המשימה הבאה.

07 ליבת המערכת · סוכני AI

Subagents — סוכני המשנה

Subagent הוא עוזר מתמחה עם חלון הקשר משלו, פרומפט מערכת משלו, סט כלים והרשאות משלו. זו אבן הבניין הראשונה של עבודה אגנטית מרובת־סוכנים — וברוב המקרים גם המועילה ביותר. הרעיון המרכזי: בידוד הקשר. במקום למלא את חלון השיחה הראשי במאות שורות פלט, אתה שולח subagent "להביא רק את התשובה".

למה זה עוצמתי כל כך

  • שמירה על ההקשר הראשי נקי — ה־subagent עושה את העבודה ה"מלוכלכת" (קריאת 30 קבצים), ומחזיר רק תקציר. ההקשר שלך נשאר חד.

2. התמקד בקבצים ששונו בלבד
3. החזר משוב לפי עדיפות: קריטי / אזהרות / הצעות

סוכן שמונחה "מצא פערים" — ימצא פערים, גם אם אין. שים לב שבדוגמה הוספנו במפורש "דווח רק על בעיות נכונות... אל תמציא". בלי זה, סוכני סקירה נוטים להציף אותך בהערות חסרות ערך.

  • בשפה טבעית — "השתמש בסוכן test-runner כדי…".

מקביליות, עלות ו־Fork

אפשר להריץ כמה subagents במקביל (חוסם בקדמת המסך, או ברקע). כל אחד מחזיר רק את התקציר שלו. אבל זהירות: הרצת הרבה סוכנים שכל אחד מחזיר פלט מפורט — מציפה את ההקשר הראשי. אפשר להגביל אילו סוגי סוכנים ניתן להפעיל בעזרת תחביר כמו Agent(worker, researcher) בשדה tools.

Subagent מפוצל (Fork) v2.1.161+ — בעזרת /fork, הסוכן יורש את השיחה המלאה במקום להתחיל נקי, ומשתמש מחדש ב־prompt cache של ההורה. זה זול יותר מ־subagent טרי כשצריך הרבה רקע.

Skills — כישורים לסוכן

Skill הוא תיקייה עם קובץ SKILL.md שמלמד את הסוכן לבצע משימה ספציפית טוב יותר — מוסכמות, נהלים, או אפילו סקריפטים מוכנים. בניגוד ל־subagent (שהוא "מי" מבצע), Skill הוא "איך" מבצעים. זהו תקן פתוח (Agent Skills) שעובד באופן זהה ב־Claude.ai, ב־Claude Code, ב־Agent SDK ובפלטפורמת המפתחים.

המבנה

תיקיית skill מכילה SKILL.md (חובה: name ו־description ב־frontmatter, ואז הוראות ב־Markdown), ואופציונלית תיקיות scripts/, references/ ו־assets/. המיקומים: .claude/skills/ (פרויקט), ~/.claude/skills/ (משתמש), וכן skills מובנים כמו /code-review ו־/debug.

.claude/skills/pdf-report/SKILL.md
---
name: pdf-report
description: יוצר דוח PDF ממותג מנתוני JSON. השתמש כשמבקשים "דוח", "PDF" או "סיכום מודפס".
allowed-tools: Read, Bash
---
# יצירת דוח PDF

1. קרא את הנתונים מהקובץ שסופק.
2. הרץ את `scripts/build.py` עם הנתיב כארגומנט.
3. הפלט נשמר ב-`output/report.pdf`.

## Gotchas (טעויות שכדאי להימנע מהן)
- אל תייצר את ה-PDF ב-pure HTML — השתמש בסקריפט המצורף.
- ודא שהפונט העברי נטען, אחרת הטקסט יופיע הפוך.

גילוי הדרגתי (Progressive Disclosure) — הקסם האמיתי

📌למה אפשר להתקין הרבה skills בזול

בתחילת הסשן נטענים רק ה־name וה־description של כל skill (זול מאוד). תוכן ה־SKILL.md המלא וקבצי העזר נטענים רק כשצריך אותם. סקריפטים רצים דרך bash בלי לצרוך הקשר כלל. כך אפשר להחזיק עשרות skills בלי לבזבז טוקנים.

הפעלה ותכונות אגנטיות

  • אוטומטית — כש־description מתאים לבקשה. כתוב תיאורים "דחפניים" — Claude נוטה לא להפעיל מספיק.
  • ישירה — /skill-name.
  • context: fork — מריץ את ה־skill בתוך subagent (אפשר לבחור agent: Explore).
  • allowed-tools — הגבלת כלים (ב־CLI; ה־SDK מתעלם).
  • disable-model-invocation: true — הופך אותו לידני בלבד.
  • $ARGUMENTS — הזרקת קלט מהמשתמש לתוך ה־skill.
  • בשדה skills: של subagent — טעינה מראש של תוכן skill מלא.
💡skill-creator והמאגר הרשמי

Anthropic מספקת meta-skill בשם skill-creator שעוזר לכתוב skills. המאגר הרשמי: github.com/anthropics/skills. החלק היקר ביותר בכל skill הוא סעיף ה־Gotchas — טעויות אמיתיות שהסוכן עושה. שפר את ה־skill על ידי צפייה איך Claude שני באמת משתמש בו.

🛠️תרגול
  1. צור skill בשם commit-msg שמלמד את הסוכן את פורמט ה־commit של הצוות שלך (Conventional Commits, למשל).
  2. הוסף לו סעיף Gotchas עם 2–3 טעויות שראית את הסוכן עושה.

MCP — חיבור הסוכן לעולם החיצון

סוכן שמוגבל לקבצים מקומיים — מוגבל בכוחו. Model Context Protocol (MCP) הוא תקן פתוח שמאפשר ל־Claude Code לדבר עם שירותים חיצוניים: GitHub, Linear, Sentry, מסדי נתונים, Playwright ועוד. Claude Code הוא הלקוח (client), ושרתי MCP חושפים כלים שהסוכן יכול לקרוא להם.

שלוש דרכי תקשורת (Transports)

▸

stdio

תהליך מקומי. הכי פשוט לשרתים שרצים על המכונה שלך.

⇅

HTTP מומלץ

לשרתים מרוחקים, עם תמיכת OAuth 2.1.

⊘

SSE מיושן

הוחלף לטובת HTTP. אל תשתמש בחדש.

הוספה וניהול

bash
# שרת מקומי (stdio)
claude mcp add --transport stdio github -- npx @modelcontextprotocol/server-github

# שרת מרוחק (HTTP)
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# ניהול
claude mcp list
claude mcp get github
claude mcp remove github

# אימות OAuth ישירות מהמעטפת (v2.1.185+)
claude mcp login sentry
claude mcp logout sentry

בתוך הסשן, /mcp מציג סטטוס ומאפשר אימות OAuth. שים לב: כל הדגלים (--transport, --scope, --env) באים לפני שם השרת, ו־-- מפריד את פקודת ה־stdio.

סקופים — ומי גובר

לפי קדימות (גבוה גובר): local ← project (קובץ .mcp.json, מומלץ ל־commit כדי לשתף עם הצוות) ← user ← פלאגין ← קונקטורים של claude.ai.

Tool Search — חיסכון טוקנים מהותי ברירת מחדל ב-2026

📌למה זה משנה

בעבר, כל הגדרות הכלים של כל שרתי ה־MCP נטענו לראש ההקשר — בזבוז עצום. כיום, במקום זאת, Claude שולף כלים לפי דרישה. במקרה אחד שתועד, עומס ה־MCP ירד מ־51K ל־8.5K טוקנים. אפשר לכבות עם ENABLE_TOOL_SEARCH=false, ולכוונן עם MCP_TIMEOUT ו־MAX_MCP_OUTPUT_TOKENS.

שרתים פופולריים

GitHubLinearSlackSentry Postgres / SupabasePlaywrightFigma Context7 (תיעוד חי)

קטלוג ה־MCP של Docker מספק 300+ שרתים מאומתים ומכולתיים עם התקנה בלחיצה.

💡זווית אגנטית: צמצם MCP ל־subagent בודד

כדי לא להעמיס הגדרות כלים על ההקשר הראשי, אפשר לחבר שרת MCP רק ל־subagent מסוים דרך השדה mcpServers ב־frontmatter שלו. כך, למשל, סוכן ה"חקירה" שלך הוא היחיד שמחזיק את כלי Sentry — וההקשר הראשי נשאר רזה.

🛠️תרגול
  1. הוסף שרת MCP של GitHub והרץ /mcp כדי לאמת חיבור.
  2. בקש מ־Claude "פתח לי issue ב־repo עם סיכום הבאג שתיקנו".
  3. צור subagent שמחזיק את שרת ה־MCP בלבד דרך mcpServers, ובדוק שההקשר הראשי לא נטען בכליו.

Hooks — אוטומציה ואכיפה דטרמיניסטית

Hooks הם מטפלים (shell, HTTP, prompt או agent) שרצים אוטומטית באירועי מחזור־חיים של הסוכן. ההבדל מ־CLAUDE.md מהותי: CLAUDE.md הוא בקשה שהסוכן עשוי לקיים; hook הוא חוק שנאכף בקוד.

חדש: hooks רואים את רמת ה־effort הפעילה דרך $CLAUDE_EFFORT, ואירוע Notification מדווח agent_needs_input / agent_completed מסוכני רקע — נוח לחיבור התראות דסקטופ.

שליטה עדינה ב־PreToolUse

פלט JSON מ־hook של PreToolUse תומך ב־permissionDecision בערכים "allow" / "deny" / "ask", וגם ב־updatedInput לשינוי הקלט לפני ההרצה. כך אפשר, למשל, לחסום אוטומטית push ל־main או פקודות rm -rf מסוכנות.

🛑זהירות מלולאה אינסופית

hook על Stop שמחזיר "המשך לעבוד" יכול להיכנס ללולאה אינסופית. הגן עליו על ידי בדיקת הדגל stop_hook_active ועצירה אם הוא כבר פעיל.

⚠️טעות נפוצה: hook-spam שאוכל טוקנים

אחד משלושת בזבזני הטוקנים הגדולים הוא הצפת פלט מ־hooks (למשל פורמטר ש"מדבר" הרבה). ודא שה־hooks שלך שקטים — מפנים פלט שגיאה רק כשבאמת יש בעיה.

🎯תרחיש אמיתי: "שער איכות" אוטומטי

שילוב חזק: PostToolUse מפעיל prettier+eslint על כל קובץ שנכתב, ו־Stop מריץ את חבילת הטסטים — ואם משהו נכשל, מחזיר את הסוכן לעבודה אוטומטית. התוצאה: קוד שעובר תמיד דרך אותו שער איכות, בלי שתצטרך לבקש.

// מפזר סוכנים במקביל — אחד לכל קובץ

);

// סוכן מאמת מנסה להפריך כל ממצא לפני האיחוד
const verified = await agent({
  prompt: "בדוק והפרך את הממצאים; השאר רק ודאיים",
  input: findings
});

return summarize(verified);

Headless, Agent SDK, CI ו-Worktrees

עד כה עבדנו אינטראקטיבית. אבל אותו מנוע אגנטי רץ גם ללא ממשק — בסקריפטים, ב־CI, וכספרייה. כאן גם מסתתרת מלכודת חיוב חשובה שחובה להכיר.

מצב Headless

bash
# הרצה לא-אינטראקטיבית עם סט כלים מוגבל
claude -p "סכם את שינויי ה-PR האחרון" \
  --allowedTools "Read,Bash(git*)" \
  --output-format json \
  --bare

claude -p "המשך מאיפה שעצרנו" --resume <session-id>

הדגל --bare מומלץ לסקריפטים (ויהפוך לברירת המחדל של -p).

Claude Agent SDK

הספרייה שהֵרֵיצה את Claude Code זמינה לבנייה עצמית — שונתה השם מ"Claude Code SDK" ל־Claude Agent SDK שם ישן הוסר (@anthropic-ai/claude-agent-sdk, ל־Python ו־TypeScript):

import { query } from "@anthropic-ai/claude-agent-sdk";

const result = query({

  1. סוכן יחיד — שיח הדוק, הקשר משותף.
  2. Subagent — עבודה מבודדת שמחזירה תקציר.
  3. Subagent משורשר/Fork — רצף שלבים, או צורך בהרבה רקע.
  4. Agent Team — עבודה שיתופית עם תלויות והודעות בין עמיתים.
  5. Dynamic Workflow — מקביליות מסיבית, עצמאית, עם אימות.
💡שלושת ההרגלים שישנו לך הכי הרבה

1. תכנן לפני שתבצע (Plan Mode). 2. נהל את ההקשר באכזריות (/clear, כלל ה־40%). 3. בודד עבודה מלוכלכת ב־subagent — וטפס במדרג רק כשהמקביליות אמיתית.

🛠️תרגול מסכם
  1. בחר פיצ'ר אמיתי. עבור עליו מקצה לקצה: /clear → Plan Mode → ביצוע → @code-reviewer בהקשר נקי → commit.
  2. הגדר hook אחד שאוכף משהו שחשוב לך (פורמט, או טסטים לפני סיום).
  3. צור subagent אחד שתשתמש בו שוב ושוב, ושמור אותו ב־.claude/agents/ כדי לשתף עם הצוות.
  4. חזור לבורר בפרק 13 והרץ אותו על המשימה הבאה שלך.
